# 屏记商业级 PRD v2.0

## 1. 文档信息

- 产品名称：屏记
- 产品定位：以锁屏可见性为核心的极简防忘事项产品
- 文档版本：v2.0
- 文档日期：2026-05-22
- 目标平台：iOS 优先，Android 仅做需求验证与等待名单
- 文档用途：用于产品、设计、研发、测试、增长与商业化协同

## 2. 执行摘要

屏记解决的不是“任务太多”，而是“重要小事明明记了，却还是忘了”。

现有替代方案主要有四类：备忘录、待办应用、聊天助手、自我通知。它们共同的问题不是不能记录，而是不能在“最该被看见的时候”稳定出现。屏记的差异化不是做更重的任务管理，而是把少量关键事项固定在用户最高频看到的位置，降低遗忘率，缩短记录到执行的路径，并用可靠、轻量、低打扰的方式建立信任。

这是一款先验证强需求、再逐步验证付费意愿的产品。MVP 只围绕一件事展开：让用户用最少操作创建一条重要事项，并让它在锁屏、小组件、提醒和系统快捷入口中持续可见，直到被完成、删除或恢复。

## 3. 背景与市场机会

### 3.1 问题现状

- 用户已经在用备忘录、微信文件传输助手、系统提醒、截图、便签等方式提醒自己。
- 这些方式最大的问题不是功能不够，而是入口重、回看弱、提醒被淹没、记录后仍然忘记看。
- 复杂的任务管理产品强调项目、标签、看板、优先级和协作，但大量普通用户只想“别忘了这件小事”。

### 3.2 市场切口

- 屏记面向的是“高频看手机、低意愿维护系统、容易被打断、需要视觉提醒”的人群。
- 这类用户不一定愿意为完整任务管理付费，但愿意为“更不容易忘”和“更顺手”付费。
- 锁屏、小组件、Action Button、快捷指令、App Intents 组合，提供了比传统待办 App 更接近日常行为流的触达面。

### 3.3 机会判断

- 用户对“轻量待办”有持续需求，但对“复杂管理”天然排斥。
- iOS 系统入口越来越多，给“少输入、多曝光”的产品提供了更好的承载位。
- 若产品先把“可靠显示”和“低成本记录”打磨到位，再叠加高级提醒、同步和多设备能力，具备形成付费层的可能。

## 4. 产品愿景与目标

### 4.1 产品愿景

成为用户在锁屏场景下最可靠、最顺手、最不容易漏看的电子便签。

### 4.2 商业目标

- 在 3 个月内验证“锁屏可见待办”是否存在稳定留存和可付费人群。
- 在 6 个月内建立免费可传播、付费可升级的基础商业闭环。
- 在 iOS 路径跑通前，不提前投入完整 Android 客户端开发。

### 4.3 产品目标

- 让用户在 3 秒内完成一条事项记录。
- 让重要事项在锁屏链路上稳定可见，不因过期、刷新失败或权限变化无故消失。
- 让用户对事项状态有明确预期，避免“误完成、误删除、误消失”。
- 让早期付费点集中在“可靠性增强、展示增强、提醒增强、多端增强”。

### 4.4 北极星指标

- 北极星指标：创建后 24 小时内至少被锁屏链路展示过一次，且最终被完成或主动处理的事项占比。

### 4.5 守护指标

- 事项异常消失投诉数
- 删除后找回失败率
- 锁屏隐私泄露反馈数
- 通知授权拒绝后的核心链路可用率

## 5. 非目标

- 不做复杂项目管理。
- 不做团队协作、评论、分配和审批。
- 不做甘特图、看板、复杂标签体系和多层级子任务。
- 不在 v1 阶段与 Todoist、滴答清单、Notion 形成全面能力竞争。
- 不在 iOS 核心体验未稳定前开发完整 Android App。

## 6. 目标用户

### 6.1 高频遗忘型用户

经常忘记取快递、带证件、回消息、付款、提交材料等。对他们来说，问题不是没有记，而是记了以后没有持续看见。

### 6.2 轻量待办用户

不愿投入精力维护复杂系统，只接受一句话输入、立即生效、无需整理的产品体验。

### 6.3 锁屏高频查看用户

一天会多次点亮手机，但不会主动打开待办 App。锁屏是他们最自然、最低成本的提醒触点。

### 6.4 视觉提醒依赖用户

比起一次性通知，更依赖“反复看见”来推动执行，对持续展示比对单次提醒更敏感。

## 7. 用户任务与典型场景

### 7.1 用户任务

- 当我想到一件不能忘的小事时，我希望立刻记下。
- 当我频繁看手机时，我希望它一直出现在能看到的地方。
- 当事情过期了，我希望它不会悄悄消失。
- 当我误删或误完成时，我希望能找回。
- 当我不想暴露内容时，我希望锁屏只显示安全信息。

### 7.2 典型场景

- 出门前：带身份证、拿伞、取快递、带药。
- 工作中：下午 4 点前发合同、回客户消息、会上补充资料。
- 睡前：明早带电脑充电器、给孩子交材料、早上出门前提醒。
- 临时记录：通过快捷入口快速记下一句提醒，不进入复杂编辑流程。

## 8. 版本策略与范围边界

### 8.1 MVP 原则

MVP 只验证一个命题：用户是否愿意把“最怕忘的小事”交给一个以锁屏展示为核心的工具。

### 8.2 MVP 必须包含

- 创建、编辑、完成、删除、恢复事项
- 锁屏/桌面小组件基础展示
- 过期事项持续保留展示
- 最近删除、最近完成记录
- App 内快速添加
- 本地通知提醒
- 基础隐私显示模式
- 稳定的排序、快照与恢复机制

### 8.3 MVP 不包含

- 团队协作
- 复杂日历视图
- 多层级子任务
- 自动 AI 建任务
- 完整 Android App
- 大而全的跨平台同步体系

### 8.4 v1 增强方向

- 系统级快捷入口完善
- 多种锁屏展示样式
- 强提醒模式
- iCloud 同步
- 付费试用与订阅能力

### 8.5 v1.5 观察方向

- Apple Reminders 导入或单向同步
- 剪贴板/分享文本智能提取
- Apple Watch 与多设备协同
- Android 小组件需求验证结果落地

## 9. 核心体验原则

### 9.1 先可见，后管理

屏记优先解决“被看见”，而不是提供完整管理能力。

### 9.2 先可靠，后丰富

任何会导致事项无故消失、误删、误完成或状态不清的能力，优先级高于新功能扩展。

### 9.3 先轻量，后个性化

输入路径必须短，默认流程必须少决策，增强选项尽量下沉。

### 9.4 隐私高于便利

锁屏、小组件、通知、快捷入口都不能为展示便利牺牲隐私安全。

## 10. 核心用户旅程

### 10.1 创建并完成事项

1. 用户打开 App 或通过快捷入口进入快速添加。
2. 输入一句事项内容，例如“明早带身份证”。
3. 可选设置时间、置顶、隐私、提醒方式。
4. 保存后事项进入 `active` 状态，并写入稳定快照。
5. 锁屏/小组件在允许的刷新窗口内更新显示。
6. 用户完成事项后主动点击完成。
7. 事项进入 `completed` 状态，并保留最近完成记录。

### 10.2 事项过期但未处理

1. 用户创建带时间的事项。
2. 到点后若未完成，事项只改变展示标签为“已过期”。
3. 事项底层状态仍保持 `active`，不会自动完成、不会自动删除。
4. 锁屏排序优先级提升，直到用户完成、编辑或删除。

### 10.3 误删恢复

1. 用户删除事项。
2. 系统执行软删除，状态改为 `deleted`，记录删除时间。
3. 用户在最近删除中恢复事项。
4. 事项回到 `active` 状态，并保留操作日志。

### 10.4 通知权限被拒绝

1. 用户拒绝通知权限。
2. App 不阻断创建、编辑、完成、删除等主链路。
3. App 通过非阻塞提示引导用户使用锁屏展示或后续开启通知。

| 过期展示 | `expired` 仅作为展示派生状态，不落库为第四状态 | P0 |

| 快照机制 | 小组件读取稳定快照，刷新失败时保留最后一次有效内容 | P0 |

| 隐私模式 | 锁屏隐藏正文，仅显示数量或模糊文案 | P1 |
| 强提醒 | 重要事项支持重复提醒或持续提醒策略 | P1 |

## 12. 关键功能规则

### 12.1 事项状态规则

- 落库状态只有 `active`、`completed`、`deleted`。
- `expired` 仅用于展示层推导，不允许作为第四种持久状态入库。
- 删除默认必须是软删除，不允许直接物理删除用户事项。
- 所有状态变更必须通过统一用例编排，并补齐操作日志。

### 12.2 锁屏展示规则

- 默认优先展示最需要被处理的事项，而非最新事项。
- 已过期但未完成事项优先级高于未来事项。
- 置顶事项优先级最高，但仍受隐私规则约束。
- 没有可展示事项时显示克制的空状态，不制造焦虑。
- 小组件刷新失败时保留最后一次稳定快照，避免空白或跳变。

### 12.3 排序规则

排序优先级建议如下：

1. 置顶且未完成事项
2. 已过期且未完成事项
3. 今日到期事项
4. 无到期但手动置顶事项
5. 最近创建的未完成事项

### 12.4 快速添加规则

- 默认只有一个核心输入框，减少进入成本。
- 时间、提醒、隐私等高级选项默认折叠。
- 系统快捷入口必须优先走最短链路，避免多层跳转。

### 12.5 隐私规则

- 锁屏、小组件、通知、系统入口默认不泄露隐私事项正文。
- 隐私事项在锁屏仅显示占位文本、数量或安全提示。
- 基础隐私能力属于信任底座，不建议作为纯付费门槛。

## 13. 信息架构与入口设计

### 13.1 App 内页面

- 首页：当前待处理事项列表 + 快速添加
- 事项详情/编辑页：修改标题、备注、时间、提醒、隐私、置顶
- 最近完成页：查看近期完成事项
- 最近删除页：查看与恢复软删除事项
- 设置页：通知、隐私、展示样式、同步、会员入口

### 13.2 系统入口

- 锁屏/桌面小组件
- 控制中心快捷入口
- Action Button
- 快捷指令/App Intents
- Spotlight/Siri 建议

### 14.2 TaskEvent

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 日志唯一标识 |
| taskId | string | 关联事项 ID |
| action | enum | `create` / `update` / `complete` / `delete` / `restore` |
| createdAt | datetime | 操作时间 |
| metadata | object | 操作补充信息 |

### 14.3 派生状态

- `expired`：`status = active` 且 `dueAt < now`
- `today`：`status = active` 且 `dueAt` 在今日内
- `visibleOnLockscreen`：符合展示规则且通过隐私过滤的稳定快照项

## 15. 通知与提醒策略

### 15.1 提醒类型

- 普通提醒：到点提醒一次
- 强提醒：按设定频率重复提醒，直到完成或手动关闭
- 展示提醒：即使通知未授权，锁屏链路仍可持续展示

### 15.2 失败降级

- 通知权限拒绝后，主链路继续可用
- Widget 刷新失败时，保留上次有效快照
- 系统快捷入口调用失败时，回退到 App 内快速添加

## 16. 商业化策略

### 16.1 免费版价值

- 建立“值得长期放在锁屏上”的基础体验
- 覆盖轻量用户的核心记录与查看需求
- 让用户先形成使用习惯，再验证升级意愿

### 16.2 Pro 付费点

- 多种锁屏/小组件展示样式
- 强提醒模式
- iCloud 同步
- 最近删除更长保留与高级恢复
- 更强的隐私展示能力
- Apple Reminders 协同
- 智能文本提取
- Apple Watch / 多设备能力

### 16.3 付费策略建议

- 采用免费版 + Pro 订阅模式
- 提供 7 天试用或有限次数体验高级能力
- 付费文案强调“更可靠、更持续、更适合重要事项”，而非“更多高级功能”

### 16.4 不建议的商业化方式

- 不建议对基础隐私能力收费
- 不建议过早引入广告
- 不建议在未验证稳定留存前做复杂套餐与年费促销实验

## 17. 增长与运营假设

### 17.1 增长假设

- 用户首次成功把事项展示到锁屏，是最关键的激活节点。
- 高频使用者会自然形成“顺手记一条”的习惯，带来次日留存。
- Pro 转化更多来自“重要事项不能漏”和“多设备/强提醒需求”。

### 17.2 关键增长动作

- 首次启动引导用户完成小组件配置
- 在完成首次记录后引导使用快捷添加
- 在出现过期事项后教育“不会消失”的可靠性价值
- 在用户形成连续使用后，再展示 Pro 升级价值

- Pro 试用开启率
- Pro 转化率

## 19. 成功标准

- 新用户首日创建率不低于 60%
- 安装后小组件或锁屏展示启用率不低于 35%
- 7 日留存不低于 20%
- 事项异常消失投诉为 0
- Pro 试用或购买转化率不低于 3%
- Android 等待名单达到 500 人前，不进入完整客户端开发

## 20. 里程碑规划

### 阶段 1：可靠性闭环

- 修复事项消失问题
- 建立稳定快照与恢复机制
- 完成最近完成、最近删除、操作日志
- 明确过期事项展示规则

### 阶段 2：锁屏体验闭环

- 完善锁屏/桌面小组件展示
- 完成快速添加主链路
- 上线基础隐私模式
- 打磨排序与空状态体验

### 阶段 3：商业化基础闭环

- 上线强提醒
- 上线 iCloud 同步
- 提供多种展示样式
- 建立 Pro 权益与试用逻辑

### 阶段 4：系统协同与增强

- 验证 Reminders 协同价值
- 验证智能文本提取价值
- 评估 Apple Watch 场景

### 阶段 5：Android 需求验证

- 建立等待名单
- 收集机型与场景样本
- 达标后再评估 Android Widget MVP

## 21. 依赖与风险

### 21.1 外部依赖

- iOS Widget / App Intents / Action Button 能力边界
- 通知权限与系统刷新策略
- iCloud 同步稳定性
- 未来付费体系接入能力

### 21.2 主要风险

- 事项无故消失会直接破坏产品信任，属于 P0 风险。
- 小组件刷新受系统限制，不能承诺实时刷新。
- 功能过多会稀释“锁屏防忘”定位。
- 若智能提取直接自动建事项，容易造成误提醒与信任损失。
- 过早进入 Android 完整开发会拖慢 iOS 核心体验验证。

### 21.3 风险应对

- 所有关键状态流转必须记录日志并可追溯
- 优先保证快照稳定而非追求刷新炫技
- 智能能力默认只做建议，不做自动落库
- Android 只做需求验证，不抢占主资源

## 22. 验收标准

- 过期事项不会自动消失。
- 删除事项可在最近删除中恢复。
- 锁屏/小组件能正确展示置顶、过期、今日事项。
- 隐私模式下锁屏不泄露正文。
- 快速添加主路径不超过 3 秒。
- 通知未授权时，核心锁屏展示仍可使用。
- 所有 P0 功能在无阻塞缺陷前不得上线推广。

## 23. 待确认事项

- 锁屏展示默认是否只展示 1 条，还是允许用户首次就选择 1 条 / 3 条布局。
- Pro 是否从首发版本就提供订阅，还是在留存验证后再开放。
- iCloud 同步是否纳入首个商业化版本，还是作为后续增强。
- Apple Reminders 协同采用导入、单向同步还是双向同步。
- Android 等待名单的收集渠道由产品内入口主导，还是外部落地页主导。

## 24. 一句话结论

屏记不是更复杂的待办工具，而是更可靠地把“不能忘的小事”固定在用户最容易看见的位置；MVP 的成败，不取决于功能多寡，而取决于它是否真的让用户少忘事、敢相信、愿意每天用。
