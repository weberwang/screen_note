## 2. 文档职责与使用方式

这份文档不是 PRD，也不是逐日开发排期。它的职责是：

- 把产品目标拆成研发阶段。
- 说明每个阶段要交付什么、先依赖什么、如何验收。
- 统一数据规则、显示层规则、原生桥接规则和阶段门禁。
- 为后续实现计划、任务拆分、测试回归和发布判断提供稳定基线。

使用方式建议：

- 产品、设计、客户端、测试在进入某阶段前，先阅读“阶段总览表”和对应阶段正文。
- 具体页面与弹层拆分以 `docs/screen-note-page-overlay-implementation-2026-05-22.md` 为补充标准。
- 显示层结构、视觉状态和组件边界以 `Pencil` 设计稿为设计源与实现标准源；视觉原则以 `docs/screen-note-visual-design-guide-2026-05-22.md` 为补充标准。

## 3. 研发目标

本阶段研发目标不是做一个完整待办系统，而是交付一个“可靠显示在锁屏上的极简提醒工具”。所有研发决策都必须优先服务以下目标：

- 3 秒内完成一条事项记录。
- 未完成事项不会因为过期、刷新失败、重启或版本升级而消失。
- 锁屏小组件能稳定展示置顶、过期和今日事项。
- 隐私模式不会在锁屏泄露正文。
- P0 功能无阻塞缺陷后才进入上架或推广。

## 4. 核心研发原则

- 可靠性优先：删除、完成、过期都必须可追踪，禁止直接物理删除用户事项。
- iOS 优先：先完成 Flutter 主 App 与 iOS WidgetKit / App Intents 链路，再验证 Android 小组件。
- 小组件快照化：锁屏小组件读取稳定快照，不直接承担复杂查询与排序逻辑。
- 过期不消失：过期是派生显示状态，不是独立持久化状态，避免状态机膨胀。
- 离线可用：MVP 默认本地优先，iCloud 同步作为 Pro 阶段能力。
- 付费不破坏信任：基础隐私、基础可靠性、普通提醒不作为付费门槛。
- 显示层设计源唯一：`Pencil` 设计稿既是显示层设计源，也是实现标准源，不允许 Flutter 代码独立发明布局、状态结构和组件边界。

## 5. 当前工程判断

当前项目是 Flutter 多端基础工程，已具备国际化、状态管理、本地数据库、小组件桥接、本地通知、代码生成和基础平台插件依赖，适合作为屏记 MVP 的基础。

当前工程在进入真实功能开发前，需要明确以下工程前提：

- 主 App、Widget、通知、快速入口三条链路必须解耦，避免页面层直接承担平台差异。
- `Pencil` 设计稿必须成为显示层的唯一设计源与标准源，避免页面、组件、弹层在实现阶段各自长歪。
- 本地数据、快照、日志、通知调度必须形成一致的应用层编排，不能散落在页面或 Provider 副作用中。
- 测试不能只停留在 `flutter analyze`；必须覆盖规则、持久化、快照和降级链路。

## 6. 阶段门禁与发布准入

### 6.1 阶段门禁

- M0 不通过，不进入任何业务开发。
- M1 不通过，不进入锁屏展示、小组件和系统入口开发。
- M2 不通过，不进入外部测试。
- M3 不通过，不对外宣称“锁屏可用”。
- M4 不通过，不对外宣称“系统级快速添加可用”。
- M5 不通过，不对外宣称“提醒可靠可用”。
- M6、M7 允许灰度或预埋，但不得反向阻塞 M0-M5 的稳定链路。

### 6.2 阻塞级问题

出现以下任一问题，必须视为阻塞进入下一阶段：

- 事项会无故消失、被覆盖或状态错乱。
- 过期事项被自动完成、自动删除或从未完成列表消失。
- Widget 刷新失败后展示空白，且没有最后有效快照兜底。
- 隐私模式下锁屏、通知或系统入口泄露正文。
- 快速添加失败时草稿丢失。
- 通知权限拒绝时主链路无法继续使用。

### 6.3 发布准入

- 第一轮内测范围建议完成 M0-M5。
- M6 只保留权益入口、配置预埋和文案边界，不强行接入复杂同步。
- M7 只保留等待名单、问卷和能力验证，不纳入核心发布时间表。

## 8. 阶段细分

### 8.1 M0：工程基线

#### 阶段目标

把空项目整理为可持续开发、可持续设计、可持续验证的产品工程。

#### 业务范围

- 不做真实事项管理。
- 不做真实锁屏展示。
- 只搭建目录、依赖、设计源、生成链路和基础壳层。

#### 开发任务清单

架构任务：

- 建立 `lib/src` 分层结构，至少包含 `app/`、`tasks/`、`history/`、`settings/`、`widget_bridge/`、`notifications/`、`shared/`。
- 建立应用入口、主题、路由骨架和错误边界。
- 明确 `domain`、`application`、`data`、`presentation` 的责任边界。

数据任务：

- 接通 `drift`、`freezed`、`json_serializable` 代码生成链路。
- 明确数据库目录、迁移命名规范和 DTO 命名规范。
- 约定 Task / TaskEvent / WidgetSnapshot 的初始建模位置。

显示层任务：

- 建立首批 `Pencil` 设计稿，至少覆盖应用壳层、首页骨架、事项卡片、输入卡片、设置分组和 Widget 预览骨架，并明确这些设计稿是显示层设计源。
- 建立 Flutter 显示层与 `Pencil` 设计源的映射规则：页面、区块、卡片、弹层、状态组件分别如何拆分。
- 建立共享视觉基础：颜色 token、字体层级、间距、圆角、基础空态与错误态。

集成任务：

- 明确 iOS Bundle ID、App Group、Widget Extension、App Intents 命名规范。
- 接通 `gen-l10n` 命令、代码生成命令、分析命令和测试命令。
- 确定基础 CI 或本地命令清单。

#### 前置依赖

- PRD 已冻结核心 MVP 范围。
- 视觉指导文档已输出核心视觉方向。
- `Pencil` 能承载本项目显示层结构、状态分支与组件设计源。

#### 测试与验收

- `rtk flutter analyze` 无阻塞问题。
- `rtk flutter test` 能稳定运行。
- `rtk flutter gen-l10n` 可成功生成本地化代码。
- 代码生成命令可成功执行。
- Flutter 应用可启动到屏记首页骨架，而不是示例页。
- 至少一批显示层标准稿已存在，并能映射到 Flutter 组件结构。

#### 风险与降级

- 若 `Pencil` 设计源缺失，不允许直接开始大规模页面开发。
- 若代码生成链不稳定，优先修复生成链，不要手写临时代码规避。
- 若 iOS 原生命名规范未确定，先冻结命名，再进入 Widget 开发。

#### 完成定义

- 工程可持续开发。
- 显示层有标准源。
- 目录、命令、依赖、国际化、代码生成全部稳定。

### 8.2 M1：P0 事项可靠性

#### 阶段目标

完成事项生命周期闭环，解决“事项消失”这一最大信任风险。

#### 业务范围

- 只解决事项可靠性，不追求复杂页面体验。
- 锁屏和系统入口可暂不接真实展示，但数据规则必须先稳定。

#### 开发任务清单

架构任务：

- 建立 Task 领域实体、TaskEvent 领域实体和仓储接口。
- 建立事项用例层：创建、更新、完成、删除、恢复、查询、排序、过期识别。
- 建立“页面只调应用层，不直接改库”的边界。

数据任务：

- 建立 Task 表、TaskEvent 表、基础 DAO 和迁移。
- 落地软删除、恢复、完成时间、删除时间、更新时间写入策略。
- 建立过期派生规则：`status == active && dueAt < now` 仅作展示派生。
- 建立排序服务：置顶、过期、今天、无截止、最近创建。
- 建立最近完成、最近删除查询。
- 建立 30 天保留规则的数据层接口。

显示层任务：

- 提供最小验证页面或调试页，用于验证创建、更新、完成、删除、恢复和查询结果。
- 验证显示层状态命名与数据状态一致，不引入 `expired` 持久状态。
- 明确后续首页、详情、历史页需要依赖的状态切片。

集成任务：

- 建立创建/更新/完成/删除/恢复后的操作日志写入。
- 建立快照刷新触发位点的占位接口，先不做真实 Widget，但确保调用点已固定。

#### 前置依赖

- M0 已完成。
- `drift`、代码生成、本地化链路已稳定。

#### 测试与验收

- 单元测试覆盖 Task 状态流转。
- 单元测试覆盖软删除、恢复、过期派生、排序规则。
- 数据库测试覆盖重启后数据仍在。
- 删除事项可出现在最近删除。
- 完成事项可出现在最近完成。
- 任意过期未完成事项不会自动消失。

#### 风险与降级

- 若排序服务规则不稳定，不允许先在 UI 层写临时排序。
- 若日志链路未通，不允许宣称“可恢复、可追踪”。
- 若迁移不稳定，优先补迁移和回归，不扩展功能面。

#### 完成定义

- 数据可靠性链路闭环。
- 事项不会无故消失。
- 删除与恢复可信。
- 日志可追踪。

### 8.3 M2：App 内 MVP

#### 阶段目标

让用户能在 App 内 3 秒记录并管理少量重要事项。

#### 业务范围

- 首页、新建、详情、最近完成、最近删除、设置入口。
- 不要求系统入口和 Widget 同步完整完成，但显示层结构必须与后续链路兼容。

#### 开发任务清单

架构任务：

- 建立首页、详情页、新建页、历史页、设置入口的路由结构。
- 建立显示层 Provider / Hook 装配方式，统一使用 `hooks_riverpod` 组织状态和副作用。
- 建立页面与弹层的交互边界：页面负责展示和导航，弹层负责轻量决策。

数据任务：

- 接通首页未完成事项列表。
- 接通详情页读取、编辑和保存。
- 接通最近完成与最近删除列表读取。
- 接通置顶、截止时间、隐私开关、提醒模式基础字段。

显示层任务：

- 根据 `Pencil` 设计稿实现首页、快速输入卡片、事项卡片、状态标签、空态、错误态。
- 根据 `Pencil` 设计稿实现新建页、详情页、最近完成页、最近删除页、设置页入口。
- 先补 `Pencil` 设计源再写代码：页面、弹层、共享组件没有设计稿时不得先写 Flutter 实现。
- 保证显示层拆分与 `docs/screen-note-page-overlay-implementation-2026-05-22.md` 的组件边界一致。

集成任务：

- 创建成功后触发快照刷新接口。
- 完成、删除、恢复后触发日志与刷新接口。
- 设置页接通通知权限状态、隐私状态和 Widget 引导状态读取。

#### 前置依赖

- M1 已完成。
- 首页、详情、历史、设置入口的 `Pencil` 设计稿已输出。

#### 测试与验收

- Widget 测试覆盖首页空态、输入、创建、完成、删除和错误提示。
- Widget 测试覆盖详情页编辑、完成、删除、恢复入口。
- Widget 测试覆盖最近完成和最近删除页面状态。
- 用户能在 3 秒内完成一条创建。
- 创建失败时输入不丢。
- 过期事项仍留在首页当前事项列表。
- 显示层结构与 `Pencil` 设计稿一致，不出现结构层级漂移。

#### 风险与降级

- 若 `Pencil` 与 Flutter 实现冲突，先更新 `Pencil` 设计源，再同步代码，不允许直接在代码层私改。
- 若新建页与首页快速创建边界不清，优先保证首页 3 秒创建链路。
- 若历史页复杂度过高，先保证恢复能力，再扩展说明文案和附加信息。

#### 完成定义

- App 内主链路闭环。
- 3 秒记录成立。
- 页面、弹层、共享组件与 `Pencil` 设计源对齐。

### 8.4 M3：iOS 锁屏小组件

#### 阶段目标

把最高优先级事项稳定展示在锁屏上。

#### 业务范围

- Widget 快照生成。
- WidgetKit 渲染。
- 隐私模式与失败降级。
- 不要求系统入口快速添加完全打通。

#### 开发任务清单

架构任务：

- 建立 `widget_bridge/` 模块，隔离 Flutter 与 iOS Widget 的数据交互。
- 明确主 App 负责排序和快照，Widget 只负责读取和渲染。
- 明确锁屏展示模式：单条、三条、今日事项。

数据任务：

- 设计 WidgetSnapshot DTO。
- 设计 App Group 共享快照结构和版本字段。
- 建立快照生成、覆盖写入、最后有效快照保留机制。

显示层任务：

- 先在 `Pencil` 输出 Widget 单条、三条、今日、隐私、空状态五套设计稿。
- Flutter 侧实现设置预览卡片时，必须与 Widget 设计稿一致。
- iOS Widget 渲染以 `Pencil` 状态分支为唯一设计源与显示标准，不单独发明布局。

集成任务：

- 主 App 在事项变更后触发快照更新。
- 主 App 请求 WidgetKit 刷新。
- Widget 读取共享快照并渲染。
- 无快照时展示空状态；读取失败时保留最后有效内容。

#### 前置依赖

- M1 已完成数据规则。
- M2 已完成基础页面和设置入口。
- Widget 标准稿已输出。
- App Group 命名和原生工程基线已确定。

#### 测试与验收

- 已过期未完成事项优先于未来事项。
- 置顶事项优先展示。
- 隐私模式下不展示正文。
- Widget 刷新失败不展示空白。
- 设置预览与真实 Widget 规则一致。
- 原生集成测试覆盖共享数据读写。

#### 风险与降级

- 若系统刷新窗口不稳定，文案不得承诺实时刷新。
- 若 Widget 读取失败，优先保留最后有效快照，不展示错误信息给用户。
- 若隐私规则无法完全保证，停止对外开放隐私模式宣传。

#### 完成定义

- 锁屏可稳定展示事项。
- 隐私与失败降级可信。
- Widget 与设置预览规则一致。

### 8.5 M4：快速添加与系统入口

#### 阶段目标

提供比打开 App 更轻的记录路径。

#### 业务范围

- App 内快速添加。
- App Intents。
- 控制中心、锁屏、Action Button 入口。
- 失败回流和草稿保留。

#### 开发任务清单

架构任务：

- 建立系统入口来源枚举和统一回流协议。
- 建立失败兜底页面 `/quick-add`。
- 明确轻入口与完整编辑页的边界。

数据任务：

- 设计快速添加草稿结构。
- 设计来源字段、默认置顶/隐私/时间策略。
- 设计失败后草稿回流与保留机制。

显示层任务：

- `Pencil` 先输出快速添加落地页、快速添加弹层和入口失败态设计稿。
- Flutter 显示层实现自动聚焦、轻量二级选项、成功反馈和失败保草稿。
- 不允许把完整编辑页强塞进系统入口流程。

集成任务：

- 接入 App Intents 快速添加。
- 验证控制中心、锁屏、Action Button 入口路径。
- 快速添加完成后触发快照刷新。
- 失败时跳回 App 并保留草稿。

#### 前置依赖

- M2 已完成首页与快速创建体验。
- M3 已完成快照刷新链路。
- 原生入口命名与路由协议已确定。

#### 测试与验收

- 系统入口创建事项默认写入未完成。
- 流程不超过 3 秒。
- 输入失败或权限不足时草稿不丢。
- 入口失败可回流到 App。
- Widget 在系统允许的窗口内更新，不承诺实时。

#### 风险与降级

- 若某个系统入口权限或系统版本受限，允许暂时关闭该入口，但必须保留 App 内快速添加主链路。
- 若系统输入 UI 不稳定，优先用 App 内快速添加页兜底。
- 若刷新不及时，统一使用“稍后更新”文案，不制造误解。

#### 完成定义

- 用户能从至少一条系统轻入口稳定创建事项。
- 失败回流与草稿保留可信。

### 8.6 M5：通知与强提醒基础

#### 阶段目标

提供普通提醒，并为 Pro 强提醒打基础。

#### 业务范围

- 通知权限。
- 普通提醒调度。
- 点击跳转。
- 删除/完成/改期后的取消与重排。
- 强提醒字段和接口预留。

#### 开发任务清单

架构任务：

- 建立 `notifications/` 模块，隔离权限读取、调度、取消、点击解析。
- 建立通知目标路由解析和事项详情跳转协议。
- 建立普通提醒与强提醒的模式边界。

数据任务：

- 为 Task 增加提醒模式、调度时间和通知标识映射策略。
- 建立改期重排、完成取消、删除取消逻辑。
- 预留强提醒策略字段，但不默认启用重复提醒。

显示层任务：

- `Pencil` 先输出通知设置页、权限说明弹层、降级提示设计稿。
- Flutter 显示层只负责说明、授权入口和降级提示，不直接调度通知逻辑。

集成任务：

- 请求通知权限。
- 调度普通提醒。
- 通知点击进入事项详情。
- 权限拒绝时展示非阻断提示，并引导继续使用锁屏展示。

#### 前置依赖

- M1 已完成 Task 数据规则。
- M2 已完成详情页与设置入口。
- M3 已完成锁屏展示主链路。

#### 测试与验收

- 到点能发送一次普通通知。
- 修改时间后通知可重排。
- 删除、完成后通知被取消。
- 点击通知可进入对应事项详情。
- 权限拒绝时用户仍可继续使用核心链路。

#### 风险与降级

- 若时区读取异常，优先降级到本地默认时区，但必须记录错误并提示稍后验证。
- 若通知权限拒绝，不阻断创建事项。
- 若重复提醒策略不稳定，禁止提前暴露强提醒真实开关。

#### 完成定义

- 普通提醒可用。
- 权限拒绝不阻断核心流程。
- 强提醒仅预留，不破坏免费主链路。

### 8.7 M6：Pro 能力与商业化预埋

#### 阶段目标

上线前完成付费边界、权益配置和可灰度能力。

#### 业务范围

- Pro 权益入口。
- 多样式、强提醒、同步等权益配置。
- 不要求所有 Pro 能力都接完真实实现。

#### 开发任务清单

架构任务：

- 建立 Pro 权益配置中心。
- 建立权益读取和显示边界，避免页面散落手写权限判断。
- 明确“入口展示”和“真实接入”是两层不同完成度。

数据任务：

- 预留会员状态、权益开关、本地缓存策略。
- 预留强提醒、多样式、同步等能力标识。
- 设计最近删除恢复、隐私增强等权益边界。

显示层任务：

- `Pencil` 先输出 Pro 页、权益卡片、付费说明弹层设计稿。
- 显示层文案必须明确哪些已可用、哪些仍在验证中。
- 不允许用付费弹窗打断 P0 创建链路。

集成任务：

- 接入购买入口或占位接口。
- 验证恢复购买、权益读取和失败降级。
- iCloud、Reminders、Watch 等能力只做技术验证时，不要默认进入主链路。

#### 前置依赖

- M2 已完成设置页结构。
- M5 已完成提醒基础能力。
- 产品已明确免费/付费边界。

#### 测试与验收

- 免费链路不被 Pro 阻断。
- 未购买用户能看到权益解释，但仍能使用基础能力。
- 购买失败不影响返回和继续使用。
- 强提醒、多样式、同步入口状态与真实接入程度一致。

#### 风险与降级

- 若权益边界模糊，优先推迟 Pro 接入，不牺牲免费链路清晰度。
- 若同步稳定性不足，不默认开启真实同步。
- 若购买链路不稳定，先保留权益说明和灰度入口。

#### 完成定义

- 商业化边界清晰。
- Pro 不影响 P0 数据可靠性与基础体验。

### 8.8 M7：智能提取与 Android 验证

#### 阶段目标

验证后续增长能力，不拖慢 iOS MVP。

#### 业务范围

- 剪贴板或分享文本提取候选。
- 人工确认后创建事项。
- Android 等待名单、问卷与桌面小组件技术评估。

#### 开发任务清单

架构任务：

- 建立“候选提取 -> 用户确认 -> 创建事项”的单向流程。
- 建立 Android 验证信息收集边界，不让其侵入 iOS 主链路。

数据任务：

- 设计候选事项 DTO。
- 设计用户确认前的临时状态与来源字段。
- 设计等待名单和问卷的最小数据结构。

显示层任务：

- `Pencil` 先输出候选确认页、Android 等待名单页设计稿。
- AI 结果只展示为建议，不允许用显示层暗示“已自动写入”。

集成任务：

- 接入文本提取规则或 AI 占位接口。
- 等待名单与问卷收集。
- Android widget MVP 技术评估，不直接承诺完整 Android App。

#### 测试与验收

- 候选提取结果不会自动落库。

#### 风险与降级

- 若 Android 需求不集中，停止继续投入客户端开发。
- 若候选确认流程过重，回到“轻量建议 + 手动编辑”。

#### 完成定义

- 增长验证链路成立。
- 不拖慢 iOS 主线。
- AI 和 Android 都停留在可控验证范围。

## 9. 显示层设计源与实现标准（Pencil）

### 9.1 角色定义

- `Pencil` 是显示层的唯一设计源，也是唯一实现标准源。
- Flutter 显示层代码是 `Pencil` 设计源的工程化落地，不是独立的第二份设计。
- `docs/screen-note-visual-design-guide-2026-05-22.md` 定义视觉原则，`Pencil` 定义页面、组件、状态和结构的具体设计源标准。

### 9.2 强约束

- 新增页面、关键区块、弹层、共享组件前，默认先补或先更新 `Pencil` 设计稿。
- Flutter 不自行发明布局、间距、层级、状态分支和组件边界。
- 若实现与设计冲突，先更新 `Pencil` 设计源，再同步代码。
- 页面、组件、弹层拆分必须与 `Pencil` 节点边界对齐，不把多个视觉节点揉成一个超大 Widget。
- 设计变更必须先落 `Pencil`，再允许进入 Flutter 显示层实现，不允许代码反向成为设计事实来源。

### 9.3 映射规则

- 页面骨架对应顶层 page frame。
- 区块组件对应可独立复用的 section/frame。
- 卡片、列表项、状态标签对应复用 widget。
- 底部弹层、对话框、提示组件对应 overlays。
- 空态、错误态、加载态必须在 `Pencil` 中有独立状态分支，不能只在代码里临时补。

### 9.4 设计源约束范围

- `Pencil` 约束页面信息层级、视觉结构、状态分支、组件边界和交互入口。
- Flutter 显示层只负责把这些设计源转成可运行代码，不负责重新定义设计。
- 若某个页面、弹层或组件缺少 `Pencil` 设计源，只允许做探索性草图，不允许进入正式实现与验收。

### 9.5 验收口径

- 结构层级是否一致。
- 视觉状态是否完整。
- 组件边界是否一致。
- 交互入口与信息顺序是否一致。
- 空态、错误态、加载态是否齐全。

## 10. 跨阶段专题

### 10.1 数据与状态约束

- 持久状态只保留 `active`、`completed`、`deleted`。
- `expired` 只作展示派生，不得落库。
- 删除只走软删除。
- 恢复必须回到 `active`。
- 所有关键状态变更都要写入 TaskEvent。

### 10.2 Widget 快照链路

执行流程：

1. 事项数据变更。
2. 应用层编排排序与快照生成。
3. `widget_bridge/` 写入稳定快照。
4. 主 App 请求刷新。
5. Widget 读取最后有效快照。

约束：

- Widget 不直接查库。
- Widget 不承担复杂排序逻辑。
- 失败时保留最后有效内容。

### 10.3 通知与时区链路

- 通知调度必须使用系统时区。
- 改期、完成、删除都要触发取消或重排。
- 权限拒绝只做降级，不阻断事项主链路。

### 10.4 国际化与文案接入

- 所有用户可见文案先写 ARB，再在显示层读取。
- 标题、空态、错误态、通知文案、弹层说明全部纳入国际化。
- 文案不得承诺系统无法保证的实时刷新。

### 10.5 测试基线与回归矩阵

单元测试重点：

- Task 状态流转。
- 过期派生。
- 排序规则。
- 删除与恢复。
- 日志写入。

Widget 测试重点：

- 首页输入与创建。
- 事项卡片状态展示。
- 详情页操作。
- 历史页恢复。
- 设置页状态读取。

集成测试重点：

- App Group 快照读写。
- Widget 刷新。
- App Intents 快速添加。
- 通知点击跳转。

回归重点：

- 重启不丢。
- 升级不丢。
- 过期不消失。
- 隐私不泄露。
- 失败有降级。

## 11. 风险升级规则

- 任何导致事项消失的缺陷直接升级为 P0 阻塞。
- 任何锁屏、小组件、通知泄露隐私正文的问题直接升级为 P0 阻塞。
- 任何快照刷新失败导致空白展示且无兜底的问题直接升级为 P0 阻塞。
- 任一系统入口导致草稿丢失的问题至少升级为 P1，若影响主入口则升级为 P0。

## 12. 推荐近期执行顺序

### 12.1 串行主链路

1. 完成 M0 工程基线。
2. 完成 M1 数据可靠性。
3. 完成 M2 App 内 MVP。
4. 完成 M3 锁屏小组件。
5. 完成 M5 通知基础。

### 12.2 可并行部分

- M2 显示层开发可与 M1 后半段数据接入并行，但前提是 `Pencil` 设计稿已存在。
- M3 Widget 原生壳层可与 M2 页面开发并行，但快照协议必须先冻结。
- M4 系统入口可在 M3 快照链路稳定后并行验证。
- M6、M7 只允许在 M0-M5 稳定后以灰度方式推进。

### 12.3 推荐第一轮内测范围

- 完成 M0-M5。
- M6 只保留权益入口和配置预埋。
- M7 只保留等待名单与问卷。

## 13. 研发结论

屏记的第一性研发目标不是“做更多待办功能”，而是“可靠地让用户看见不能忘的事”。研发拆分必须先把数据可靠性、过期保留、软删除恢复、锁屏快照、小组件隐私和失败降级做扎实，再扩展强提醒、同步、AI 和 Android。只要 M0-M5 中任一条 P0 链路没有完全稳定，就不应把研发重心转向复杂管理功能、过度商业化或多平台扩张。
