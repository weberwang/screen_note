# 屏记 Flutter 全局技术基线 RD

## 0. 文档信息

- 文档类型：全局技术基线
- 输入来源：`docs/screen-note-prd-2026-05-22.md`
- 目标阶段：`technical_baseline_ready`
- 适用范围：MVP 到实施前边界
- 结论摘要：采用本地优先、按功能 DDD 分层、以稳定快照和状态可追溯为核心的 Flutter 单仓工程方案

## 1. PRD 需求理解

### 推荐方案

- 产品目标：用最短输入路径创建重要事项，并让事项在锁屏、小组件、提醒和主应用中持续可见，直到被显式处理。
- 目标用户：高频看手机、低意愿维护复杂待办系统、依赖视觉重复曝光推动执行的 iOS 用户。
- 核心场景：快速记一条、在首页持续看见、过期不消失、误删可恢复、隐私模式下仍可安全显示。
- 范围内功能：事项增删改查、完成/恢复、最近完成、最近删除、锁屏/桌面小组件、通知提醒、稳定排序与快照、设置与隐私。
- 明确不做：复杂项目管理、团队协作、多层子任务、完整 Android 客户端、首期强依赖云端同步。

### 备选方案

- 先做云同步优先的多端任务系统，再回补锁屏展示。

### 取舍理由

- PRD 的核心价值是“看得见且不消失”，不是“跨端协同”。先把本地可靠性闭环做稳，能更直接验证产品假设。

## 2. 假设前提与待确认项

### 假设前提

- MVP 仅支持单用户本地数据，不引入登录、账号体系或远端 API。
- 锁屏/小组件展示以系统允许的刷新窗口和稳定快照为边界，不承诺实时刷新。
- 通知、Widget、系统快捷入口均属于可降级能力，主链路永远以应用内创建、编辑、完成、删除、恢复为准。
- iCloud 同步、Reminders 协同、Pro 订阅能力不进入首轮实施目标集，只在设置与数据边界中预留扩展位。

### 待确认项

- 锁屏默认布局是否固定单条，还是允许首次即选择单条/三条。
- Pro 是否首发上线，还是在留存验证后启用。
- iCloud 同步是否进入首个商业化版本。
- Apple Reminders 未来采用导入、单向同步还是双向同步。
- Android 等待名单由 App 内入口还是外部落地页承接。

## 3. 业务流程与页面范围

### 推荐方案

- 主路径：首页快速添加/新建页录入 -> 事项落库为 `active` -> 应用层写操作日志 -> 快照刷新 -> Widget/锁屏展示。
- 完成路径：用户主动完成 -> 应用层更新状态为 `completed` -> 取消相关提醒 -> 写入最近完成。
- 删除路径：用户主动删除 -> 应用层软删除为 `deleted` -> 取消提醒 -> 写入最近删除。
- 恢复路径：最近删除恢复 -> 状态返回 `active` -> 重建排序与快照 -> 视需要重建提醒。
- 过期路径：仅展示派生为 `expired`，底层仍保持 `active`。

### 备选方案

- 页面直接改库，再由各入口各自补通知与快照。

### 取舍理由

- 事项状态、日志、快照、提醒存在强一致联动，必须通过应用层用例统一编排，避免页面和系统入口各写一套副作用。

### 页面/模块粗范围

- 根壳层：启动、路由、Tab/Shell、全局 Provider、主题、本地化。
- 任务主流程：首页、快速添加、事项详情/编辑。
- 历史中心：最近完成、最近删除。
- 小组件桥接：稳定快照生成、隐私过滤、Widget 展示样式。
- 设置中心：通知、隐私、Widget 样式、未来同步/Pro 入口。

## 4. 总体技术架构

### 推荐方案

- 架构形态：`app/ + shared/ + features/<feature>/` 的 DDD by feature 结构。
- 分层原则：`presentation -> application -> domain -> data/infrastructure`，页面只负责输入、展示与导航。
- 数据源：本地优先，以 `drift` 为结构化事实源，以 `shared_preferences` 存轻量偏好，以 Widget 快照作为展示投影。
- 系统能力：`home_widget`、`flutter_local_notifications`、`timezone`、`flutter_timezone` 由边界服务统一封装。
- 可靠性核心：所有关键状态变更均写事件日志，快照永远从稳定查询结果构建，不让 Widget 直接承载复杂推导。

### 备选方案

- 以页面为中心的平铺目录 + 直接在页面中调用本地存储与系统插件。

### 取舍理由

- 屏记的复杂度不在视觉，而在状态一致性与系统能力降级。DDD 分层更适合承载“状态流转 + 日志 + 快照 + 提醒”的统一编排。

## 5. Flutter 客户端详细技术方案

### 5.1 项目结构

### 推荐方案

- `lib/app`：入口、bootstrap、router、全局错误映射、ProviderScope。
- `lib/shared`：通用主题、i18n、时间格式化、通用异常、日志门面。
- `lib/features/app_shell`：壳层导航与全局启动分发。
- `lib/features/task_flow`：事项列表、快速添加、编辑、状态流转。
- `lib/features/history_center`：最近完成、最近删除、恢复入口。
- `lib/features/widget_bridge`：稳定快照、Widget 数据桥接、隐私投影。
- `lib/features/settings_center`：通知/隐私/展示偏好与未来权益入口。

### 备选方案

- 所有功能平铺在 `lib/` 根目录下。

### 取舍理由

- 后续会接入 Widget、通知、设置、同步等不同能力面，按功能收口能明显降低跨页面改动面。

### 5.2 状态管理与依赖组织

### 推荐方案

- 采用 `flutter_riverpod + hooks_riverpod`，初始化阶段补齐 `riverpod_annotation + riverpod_generator`。

### 备选方案

- 保持纯手写 Provider。

### 取舍理由

- 项目规则要求优先使用注解链路；手写 Provider 可作为过渡，但不应成为长期基线。

### 5.3 路由与守卫

### 推荐方案

- 使用 `go_router` 统一管理首页、编辑页、最近完成、最近删除、设置与系统回流入口。

### 备选方案

- 原生 `Navigator` 手写跳转。

### 取舍理由

- 多入口回流、Widget 点击、快捷入口回调需要稳定的路由宿主与根重定向策略，`go_router` 更直接。

### 5.4 本地存储与快照

### 推荐方案

- `drift` 作为任务、任务事件日志、快照元数据的唯一结构化持久层。
- `shared_preferences` 存储布局样式、隐私开关、引导状态等轻量配置。
- Widget 快照单独投影，不直接暴露主库查询给 Widget。

### 备选方案

- 全量使用 `shared_preferences` 或散落 JSON 文件。

### 取舍理由

- 屏记需要排序、过滤、恢复、事件追溯和派生展示，关系型查询能力比 KV 更适配。

### 5.5 通知、时区与系统能力降级

### 推荐方案

- 使用 `flutter_local_notifications + timezone + flutter_timezone` 统一调度提醒。
- 未授权通知时仅降级为无通知，不影响主链路。
- Widget 刷新失败时保留上一份有效快照。

### 备选方案

- 由页面零散调用通知插件。

### 取舍理由

- 提醒、删除、恢复、完成存在统一取消/重建逻辑，必须由应用层协调。

### 5.6 国际化、模型与代码生成

### 推荐方案

- 国际化：`flutter_localizations + intl + gen_l10n`
- 模型：`freezed + freezed_annotation`
- DTO/JSON：`json_annotation + json_serializable`
- 生成入口：`build_runner`

### 备选方案

- 手写文案常量与手写模型。

### 取舍理由

- 项目规则明确要求国际化和注解生成优先，先统一基线可避免后续返工。

## 6. 第三方包选型与最佳搭档

### 能力域：状态管理与依赖组织

- 推荐主方案：`flutter_riverpod + hooks_riverpod + riverpod_annotation + riverpod_generator`
- 推荐理由：同时满足显示层 Hook 能力与项目要求的 Provider 注解链。
- 最佳搭档：`go_router`、`freezed`
- 备选方案：仅手写 Riverpod Provider
- 避免混搭：不要再引入第二套全局状态框架
- 接入注意事项：初始化阶段补齐 annotation 依赖并统一生成命令
- 升级/维护风险：主要风险是生成文件未同步，需纳入 CI 校验

### 能力域：路由与系统回流

- 推荐主方案：`go_router`
- 推荐理由：适合首页、设置、历史页与 Widget/快捷入口回流的统一宿主
- 最佳搭档：Riverpod 重定向状态
- 备选方案：`Navigator` 手写封装
- 避免混搭：不要并行维护第二套路由状态机
- 接入注意事项：统一定义根路由与深链参数解析
- 升级/维护风险：集中在重定向逻辑失控，需保持单一入口

### 能力域：本地数据与恢复

- 推荐主方案：`drift + sqlite3_flutter_libs + path_provider + path`
- 推荐理由：支持任务、事件、最近删除、快照元数据的结构化查询
- 最佳搭档：Repository + 应用层用例
- 备选方案：纯 `shared_preferences`
- 避免混搭：不要再引入第二主数据库
- 接入注意事项：把 `expired` 保持为查询派生，不落库
- 升级/维护风险：迁移脚本需纳入测试

### 能力域：Widget 与提醒

- 推荐主方案：`home_widget + flutter_local_notifications + timezone + flutter_timezone`
- 推荐理由：覆盖 MVP 所需展示链路与本地提醒链路
- 最佳搭档：稳定快照服务、权限降级门面
- 备选方案：仅做 Widget，不接提醒
- 避免混搭：不要让多个页面直接调用插件实例
- 接入注意事项：权限失败、刷新失败必须以降级策略收口
- 升级/维护风险：平台插件配置与真机验证成本较高

### 能力域：模型、序列化与国际化基础

- 推荐主方案：采用默认通用基础栈，不引入额外替代体系
- 推荐理由：与现有 `pubspec.yaml` 和项目规则完全一致
- 最佳搭档：`build_runner`
- 备选方案：手写模型和字符串
- 避免混搭：不要同时混用手写模型与 `freezed` 长期共存
- 接入注意事项：新增用户可见文案必须先进 ARB
- 升级/维护风险：生成链版本需与 Flutter SDK 兼容

## 7. 后端协作与接口契约要求

### 推荐方案

- MVP 不引入远端后端，接口契约仅针对系统插件边界与未来同步预留。
- 时间规则：所有任务时间以本地时区存储 UTC 时间戳并配合系统时区展示。
- 幂等规则：完成、删除、恢复等写操作以任务 ID + 操作时间驱动日志，重复触发需安全去重。
- 错误规则：插件调用失败必须映射为可恢复领域错误，不能直抛到底层页面。

### 备选方案

- 先引入云端 API 和同步协议。

### 取舍理由

- 现阶段没有网络硬需求，引入远端只会扩大交付面与失败模式。

## 8. 数据与安全方案

### 推荐方案

- 隐私事项正文不进入锁屏、小组件、通知和系统快捷入口的明文显示。
- 日志记录操作上下文，不记录隐私正文原文。
- 最近删除保留软删除记录，恢复时只改业务状态，不做物理重建。
- 快照层只输出展示最小必要字段。

### 备选方案

- 快照与通知直接复用任务全文。

### 取舍理由

- PRD 明确“隐私高于便利”，需要从数据边界而非 UI 文案层面落实。

## 9. 埋点、监控与运营支持

### 推荐方案

- MVP 先预埋事件位，不强制接入远端分析 SDK。
- 事件分类最少覆盖：首次创建、首次启用 Widget、完成事项、恢复事项、通知权限拒绝、快照刷新失败。
- 本地日志门面统一记录关键操作结果，便于后续接入 Crash/Analytics。

### 备选方案

- 首轮即接入完整远端埋点与远程配置。

### 取舍理由

- 当前实施重点是可靠性闭环，不应让远端运营栈稀释实现重心。

## 10. 测试与质量保障

### 推荐方案

- 单元测试：任务状态流转、排序、过期派生、恢复、快照过滤、提醒编排。
- 装配测试：Drift 仓储、快照服务、通知调度降级。
- Widget/集成测试：快速添加主路径、最近删除恢复、设置切换、空状态。
- 真机校验：Widget 刷新、通知权限拒绝、时区变化、锁屏隐私模式。

### 备选方案

- 以人工真机回归为主。

### 取舍理由

- P0 风险集中在状态与系统能力联动，必须用自动化覆盖关键规则。

## 11. 实施计划与里程碑

### 推荐方案

1. 壳层初始化：项目结构、Router、主题、i18n、全局 Provider。
2. 任务主流程：任务聚合、列表、快速添加、编辑、状态流转、事件日志。
3. 历史与恢复：最近完成、最近删除、恢复与日志回放。
4. Widget/快照：稳定查询、隐私过滤、样式偏好、刷新容错。
5. 提醒与设置：本地通知、时区、权限降级、通知/隐私/展示偏好。

### 备选方案

- 先落 Widget，再补任务主流程。

### 取舍理由

- 没有稳定的任务主流程与快照事实源，Widget 只会成为不稳定的展示壳。

## 12. 风险清单与替代方案

- 技术风险：Widget 刷新受系统限制。
  - 替代方案：稳定快照优先，弱化“实时”承诺。
- 协作风险：未来引入同步与 Pro 时可能冲击当前边界。
  - 替代方案：在设置和仓储接口层预留扩展口，不提前实现。
- 体验风险：首页与历史页若共享组件边界不清，后续改动会放大成本。
  - 替代方案：模块拆分时明确共享组件与模块私有组件。
- 发布风险：通知权限或时区适配异常导致提醒不稳定。
  - 替代方案：把提醒视为增强能力，主链路不依赖提醒成功。

## 13. Handoff Notes For Module Splitting

- `app-shell` 必须单独成模块，承载根路由、Tab/Shell、系统回流与全局样式。
- 任务列表、快速添加、编辑共享同一用户任务与数据生命周期，应合并为 `task-flow`。
- 最近完成与最近删除共享“历史浏览/恢复”工作流，可合并为 `history-center`。
- Widget 快照与展示应独立为 `widget-bridge`，避免被页面逻辑污染。
- 设置页独立为 `settings-center`，承载通知、隐私、样式与未来权益入口。

## 14. 最终结论

### 推荐方案

- 采用“本地优先 + DDD 按功能分层 + 应用层统一编排副作用 + Widget 稳定快照投影”的 Flutter MVP 实施路线。

### 备选方案

- 采用更轻的页面式实现，先追求快速可用。

### 取舍理由

- 屏记的核心不是把页面做出来，而是让状态、快照、提醒与隐私边界长期可靠。当前推荐方案更符合产品主命题，也更贴合仓库既有依赖与项目规则。
