## 1. PRD 需求理解

- 产品目标：把少量重要待办稳定显示在锁屏高频触点上，保证“不丢、不误消失、可快速记录”。
- 目标用户：健忘型用户、轻量待办用户、锁屏高频用户、依赖视觉提醒的用户。
- 核心场景：快速新建事项、锁屏持续展示、过期保留、误删恢复、系统快捷入口补录、到点提醒、隐私保护展示。
- 范围内功能：事项增删改查、状态流转、最近完成与最近删除、锁屏小组件、快速添加、本地通知、基础隐私模式。
- 明确不做：复杂项目管理、团队协作、完整 Android 客户端、AI 自动写入事项、重型多端同步平台。
- 当前仓库现状：已存在 Flutter 工程、`tasks/history/settings/quick_add/widget_bridge` 五类 feature、`drift` 本地存储、`home_widget` 桥接、小组件快照与 `go_router` 路由壳层。

## 2. 假设前提与待确认项

### 假设前提

- 首个可交付版本以单用户、本地优先、无登录主链路为前提。
- iOS 是首发质量基线，Android 先保持结构兼容与验证空间，不承诺同等系统集成深度。
- 事项主链路在无通知权限、无 Widget 权限、Widget 刷新失败时仍可完成创建、编辑、完成、删除、恢复。
- `expired` 只作为展示派生状态，不作为持久化第四状态。
- 当前 `app.pen` 代表候选设计源，但尚未完成正式设计冻结与状态覆盖验收。

### 待确认项

- 首发是否必须同时覆盖控制中心、锁屏快捷入口与 Action Button，还是允许分批上线。
- 强提醒的重复频率、静音策略、夜间策略与用户自定义粒度是否进入首发。
- Pro 付费、iCloud 同步、Apple Reminders 同步、AI 提取分别计划落在哪个版本波次。
- 首发是否要求接入生产级埋点、崩溃监控与远程开关，还是允许先以内测配置交付。
- `app.pen` 是否已包含所有核心页面状态、异常态与隐私态；若未覆盖，需要先补齐再冻结。

## 3. 业务流程与页面范围

### 推荐方案

- 主路径一：用户在首页或快速添加入口输入事项，应用层统一创建事项与事件日志，随后刷新稳定快照并尽力触发 Widget/通知更新。
- 主路径二：用户完成事项后，应用层统一写入 `completed` 状态与日志，历史页可立即查看，Widget 快照同步刷新。
- 主路径三：用户删除事项时只做软删除，记录 `deletedAt` 与操作日志，最近删除页支持恢复。
- 异常路径一：通知权限拒绝时，只展示非阻断提示，不影响事项主链路。
- 异常路径二：Widget 刷新失败时，保留最后一次有效快照，不回退业务数据。
- 页面范围保持全局粗粒度，不在本阶段拆成模块实施文档：
  - 首页任务列表与快速录入
  - 事项详情 / 编辑
  - 最近完成
  - 最近删除
  - 设置 / 隐私 / Widget 展示设置
  - 系统快捷添加回流页

### 备选方案

- 先以“仅首页 + 简化编辑弹层”的极简 MVP 交付，再逐步补历史、设置和系统入口。

### 取舍理由

- 推荐方案更符合 PRD 中“可靠、不丢、持续看见”的主承诺，避免因过度精简导致状态流转和恢复链路不完整。
- 备选方案上线更快，但会削弱“误删恢复、历史可追踪、设置可解释”的信任底座。

## 4. 总体技术架构

### 推荐方案

- 采用 `lib/app + lib/shared + lib/features/<feature>` 的按业务边界分层架构。
- 应用总体形态采用“本地优先 + 系统能力桥接 + 未来远端可插拔网关”的单体客户端架构：
  - `app/`：应用壳层、Bootstrap、Router、全局装配
  - `shared/`：主题、基础组件、格式化、跨 feature 通用能力
  - `features/`：按业务边界收口任务、历史、快速添加、设置、Widget 桥接等能力
- 领域规则固定下沉到 UseCase / Service：
  - 状态流转只能通过应用层用例统一编排
  - 任何关键状态修改必须补事件日志
  - Widget、通知、系统入口只消费稳定快照或受控 DTO，不直接承载复杂推导
- 未来同步、AI、Reminders、会员等扩展通过新增 `gateway/repository` 边界接入，不把第三方 SDK 直接暴露到页面层与领域层。

### 备选方案

- 扁平 MVVM 或页面直连 Repository。
- 过早引入“客户端 + 云同步后端”双主系统架构。

### 取舍理由

- 推荐方案与当前仓库结构、项目约束以及本地优先产品定位一致，能先把可靠性主链路做实，再为 P1 同步能力预留扩展点。
- 扁平 MVVM 初期更快，但会把状态流转、日志写入、系统桥接逻辑散落到页面层，后续重构代价高。
- 过早引入远端主架构会抬高登录、鉴权、冲突处理、灰度发布与运维复杂度，与当前 PRD 的 MVP 重心不匹配。

## 5. Flutter 客户端详细技术方案

### 5.1 项目结构与边界

#### 推荐方案

- 继续巩固现有目录基线：
  - `lib/app`：应用入口、路由、全局 Provider 装配
  - `lib/shared`：主题、通用组件、基础工具
  - `lib/features/tasks`
  - `lib/features/history`
  - `lib/features/quick_add`
  - `lib/features/settings`
  - `lib/features/widget_bridge`
- 每个 feature 内保持 `domain / application / data / presentation` 的稳定分层。

#### 备选方案

- 把历史、快捷添加、Widget 桥接继续混入 tasks 大 feature。

#### 取舍理由

- 推荐方案更适合持续演进系统入口、隐私模式、提醒调度与后续同步，不会把任务域越堆越重。

### 5.2 状态管理与副作用

#### 推荐方案

- 以 `hooks_riverpod + flutter_riverpod` 为唯一页面状态与依赖注入方案。
- 在现有基线上补齐 `riverpod_annotation + riverpod_generator`，后续 Provider、Notifier、异步装配统一走注解生成。
- 页面只负责展示、输入、导航与订阅状态；副作用通过应用层服务和 Riverpod Provider 触发。

#### 备选方案

- 继续手写 Provider 装配。
- 切换到 `bloc/flutter_bloc`。

#### 取舍理由

- 当前仓库已落地 Riverpod 体系，补齐注解生成比切栈更稳，也更符合项目约束。
- 手写 Provider 在中短期可用，但会持续放大样板、依赖装配和测试维护成本。

### 5.3 路由与系统入口回流

#### 推荐方案

- 继续用 `go_router` 作为唯一路由方案。
- 系统快捷入口、通知点击、Widget 点击统一回流到 `go_router` 路由树，由应用层决定进入详情、编辑还是快速补录。

#### 备选方案

- 引入 `auto_route` 做强类型路由生成。

#### 取舍理由

- 当前仓库已存在 `go_router`，继续沿用可保持低迁移成本。
- `auto_route` 在超大路由图中更强，但当前产品体量不足以支撑额外迁移收益。

### 5.4 数据模型、序列化与代码生成

#### 推荐方案

- `domain` 与 `data` 模型统一采用 `freezed + json_serializable`。
- `drift` 负责本地结构化表、DAO 与迁移。
- 所有生成链路统一走 `build_runner`。

#### 备选方案

- 手写实体与 JSON 解析。
- 使用 `equatable` + 手写 copyWith。

#### 取舍理由

- 推荐方案与仓库既有依赖一致，能稳定支撑状态不可变、值相等、DTO 映射与数据库模型演进。
- 手写样板在任务事件、快照 DTO、设置状态增多后会快速失控。

### 5.5 本地存储、快照与系统桥接

#### 推荐方案

- `drift` 存业务主数据：事项、事件日志、最近删除所需字段、快照元信息。
- `shared_preferences` 只存轻量偏好、快速添加草稿、非敏感开关。
- `home_widget` 只写稳定快照文本、展示模式、必要 ID，不直接暴露完整私密正文给系统侧读取。
- 隐私事项在 Widget、通知、系统快捷入口默认降级为摘要化文案。

#### 备选方案

- 用 `shared_preferences` 直接存业务主数据。
- Widget 侧每次直接查询业务库或重新执行排序逻辑。

#### 取舍理由

- 推荐方案能把业务真相源与系统展示副本分开，减少 Widget 刷新失败和排序漂移带来的不一致风险。
- 备选方案开发更快，但会放大数据损坏、推导重复、平台桥接泄露和测试困难。

### 5.6 通知、时区、权限与降级

#### 推荐方案

- 本地提醒统一通过 `flutter_local_notifications + timezone + flutter_timezone` 调度。
- 通知权限、Widget 刷新、系统入口失败都按“能力降级”处理，不能阻断事项主链路。
- 过期事项只改变展示状态，不自动完成、不自动删除。

#### 备选方案

- 自行封装原生定时能力或在页面层分散调度。

#### 取舍理由

- 推荐方案复用成熟插件组合，便于跨平台差异收口、测试替身和统一调度策略。
- 原生分散实现初期看似灵活，但长期会引入平台行为不一致和重复维护成本。

### 5.7 国际化、主题与隐私默认值

#### 推荐方案

- 继续使用 `flutter gen-l10n + intl + flutter_localizations`。
- 用户可见文案一律进 ARB，不新增长期硬编码字符串。
- 主题层继续采用 `shared/presentation/theme` 收口，等待后续设计冻结后再做 Token 对齐。
- 隐私规则默认高于展示便利：锁屏、小组件、通知、系统快捷入口都不能暴露私密事项正文。

#### 备选方案

- 首版只做中文硬编码，后续再补国际化。

#### 取舍理由

- 当前仓库已接入本地化生成链路，继续保持比后补低成本，也更利于 Widget/通知文案的一致性。

## 6. 第三方包选型与最佳搭档

### 能力域：状态管理与依赖组织

- 推荐主方案：`flutter_riverpod + hooks_riverpod + riverpod_annotation + riverpod_generator`
- 推荐理由：与现有仓库一致，并满足项目对注解生成与展示层副作用治理的要求。
- 最佳搭档：`go_router`、应用层 UseCase、`build_runner`
- 备选方案：`bloc/flutter_bloc`
- 避免混搭：不要再引入第二套全局状态框架，不要并行维护手写单例容器
- 接入注意事项：需要把后续 Provider/Notifier 迁到注解生成链路
- 升级/维护风险：Riverpod 体系升级时要同步检查 generator/annotation 兼容性

### 能力域：路由与系统入口

- 推荐主方案：`go_router`
- 推荐理由：当前已落地，足以覆盖深链、通知点击与系统入口回流
- 最佳搭档：`flutter_riverpod`
- 备选方案：`auto_route`
- 避免混搭：不要同时存在第二套路由框架或页面内手写守卫状态机
- 接入注意事项：系统入口参数解析必须收口到统一路由与应用层
- 升级/维护风险：路由参数模型增多后要补 typed wrapper，避免字符串散落

### 能力域：本地结构化存储

- 推荐主方案：`drift + sqlite3_flutter_libs + path_provider + path`
- 推荐理由：支持结构化查询、软删除、事件日志、迁移与稳定快照元信息
- 最佳搭档：`freezed`、Repository、UseCase
- 备选方案：`isar`
- 避免混搭：不要把 `shared_preferences` 当主数据库，不要并存多套主数据库
- 接入注意事项：所有状态流转必须通过应用层后再落库并写日志
- 升级/维护风险：表结构扩展时要同步维护迁移与回归测试

### 能力域：轻量偏好与草稿

- 推荐主方案：`shared_preferences`
- 推荐理由：适合展示模式、引导状态、快速添加草稿等轻量配置
- 最佳搭档：feature 内部存储驱动与 Repository 边界
- 备选方案：全部收进 `drift`
- 避免混搭：不要把复杂业务状态存入键值对
- 接入注意事项：仅允许存非敏感、可容忍覆盖的轻量数据
- 升级/维护风险：键名演进要集中管理，避免页面层直接读写

### 能力域：Widget 与系统桥接

- 推荐主方案：`home_widget`
- 推荐理由：当前产品强依赖锁屏与桌面展示，这是最直接的 Flutter 桥接方案
- 最佳搭档：稳定快照构建器、App Group 共享存储、隐私文案解析器
- 备选方案：完全自写平台通道
- 避免混搭：不要让多个页面直接调用 `HomeWidget`；不要让 Widget 侧自行重算业务排序
- 接入注意事项：桥接层只读稳定快照，不直接暴露任务全量正文
- 升级/维护风险：iOS WidgetKit 与 Android Widget 能力差异要持续隔离在桥接层

### 能力域：通知与时区

- 推荐主方案：`flutter_local_notifications + timezone + flutter_timezone`
- 推荐理由：覆盖到点提醒、跨时区调度和能力降级
- 最佳搭档：应用层提醒调度服务、权限说明页
- 备选方案：平台原生分散实现
- 避免混搭：不要同时维护多套提醒调度入口
- 接入注意事项：提醒失败只影响提醒，不影响任务真相源
- 升级/维护风险：夏令时、时区切换与重复提醒规则需要专项回归

### 能力域：通用基础组合

- 推荐主方案：采用“简化后的通用基础栈”
- 推荐理由：当前产品是本地优先 MVP，应保留 `gen_l10n + intl + freezed + json_serializable + build_runner`，但不默认引入 `fpdart`
- 最佳搭档：`flutter_localizations`、`freezed_annotation`、`json_annotation`
- 备选方案：进一步简化为手写模型与硬编码文案
- 避免混搭：不要在同一项目里并存多套 i18n 与模型生成方案
- 接入注意事项：后续 Provider 注解生成应补齐到这套基础栈中
- 升级/维护风险：代码生成链失配会直接影响 CI 与本地开发体验

### 能力域：未来网络与同步

- 推荐主方案：首版不把 `dio + retrofit` 接入运行时主链路，只在全局基线中预留接口边界
- 推荐理由：当前 PRD 的 P0 核心是本地可靠性与锁屏展示，不应提前为尚未确认的云同步与 AI 能力引入额外复杂度
- 最佳搭档：未来若接入远端，再采用 `dio + retrofit + json_serializable + freezed`
- 备选方案：现在就落地 `dio + retrofit`
- 避免混搭：不要同时引入 `http`、`dio`、平台通道请求三套网络入口
- 接入注意事项：一旦进入 iCloud、自建后端或 AI 服务阶段，统一通过 Gateway/Repository 接入
- 升级/维护风险：延后接入网络会推迟真实联调，但能显著降低当前 MVP 的工程噪音

## 7. 后端协作与接口契约要求

### 推荐方案

- MVP 主链路不依赖业务后端，不把“上线前必须有云端”设为前提。
- 为未来扩展保留统一接口契约规则：
  - 鉴权模型：当前无登录；若后续引入同步或会员，鉴权状态必须独立于任务主链路，可降级为本地模式
  - 错误结构：统一采用 `code + message + details + traceId`
  - 时间规则：服务端一律 UTC ISO8601；客户端保存原始时间戳并结合本地时区做展示与提醒
  - 幂等要求：任务创建、更新、删除、恢复、事件补写都必须支持幂等语义
  - 分页规则：历史记录与日志若未来远端化，默认基于游标分页
  - 实时边界：首版不要求实时推送，提醒由本地通知承担

### 备选方案

- 现在就设计完整后端、登录态、远端日志与同步协议。

### 取舍理由

- 推荐方案更符合 PRD 的 MVP 重心，同时为 P1 的同步、Pro、AI、Reminders 留足协作契约。
- 过早做完整后端契约会抬高跨端、运维和合规成本，而首版收益不明显。

## 8. 数据与安全方案

### 推荐方案

- 数据真相源：
  - `drift`：事项、事件日志、必要快照元数据
  - `shared_preferences`：轻量设置、引导状态、非敏感草稿
- 安全边界：
  - 私密事项正文默认不出现在锁屏、小组件、通知与系统快捷入口
  - 日志、埋点、诊断输出必须对私密事项做脱敏或只上传类型标签
  - 删除默认软删除，不允许直接物理删除
- 未来敏感凭据：
  - 当同步、会员或 AI 密钥真正进入交付范围时，再引入 `flutter_secure_storage`
  - 该方案已在官方 pub.dev 页面验证可用于平台安全存储，适合作为后续凭据落点：[flutter_secure_storage](https://pub.dev/packages/flutter_secure_storage)

### 备选方案

- 直接把所有配置和主数据都放进本地 SQLite 或 preferences
- 现在就提前引入密钥存储依赖

### 取舍理由

- 推荐方案能保持当前依赖简洁，同时不牺牲未来安全演进路径。
- 过早引入未使用的安全存储会增加平台配置复杂度；完全不预留又会让后续同步接入成本上升。

## 9. 埋点、监控与运营支持

### 推荐方案

- 事件体系围绕 PRD 成功指标设计：
  - 页面事件：首页曝光、历史页曝光、设置页曝光、快速添加入口曝光
  - 行为事件：创建、编辑、完成、删除、恢复、设置提醒、切换隐私模式、切换 Widget 展示模式
  - 漏斗事件：安装后首次创建、首次启用 Widget、首次完成事项、首次恢复删除事项
- 监控策略：
  - 首版至少保留可插拔埋点接口和结构化日志边界
  - 崩溃与性能监控在准备正式对外测试或上架前接入 `Crashlytics` 或 `Sentry`
  - 远程开关、A/B 与灰度首版不强制，但需为高风险系统入口保留本地降级开关

### 备选方案

- 首版完全不做埋点和监控
- 一开始就接入全套运营平台

### 取舍理由

- 推荐方案避免“客户端做完了但无法判断真实可用性”的盲飞状态，也不会把首版拉入过重运营体系。

## 12. 风险清单与替代方案

### 推荐方案

- 技术风险：Widget 刷新受系统限制，必须坚持“稳定快照 + 保留最后有效内容”策略。
- 设计风险：`app.pen` 若未覆盖异常态和隐私态，后续实现很容易出现设计解释分叉。
- 协作风险：若不先完成技术基线与设计冻结，模块拆分和实现会继续基于口头约定漂移。
- 发布风险：通知、Widget、App Group、系统入口能力高度依赖平台配置，必须纳入发布前检查单。
- 扩展风险：若未来同步直接侵入现有本地逻辑而不经 Repository/Gateway，会导致双真相源冲突。

### 备选方案

- 忽略上游冻结，按现有代码继续推进。
- 为未来能力一次性引入全量云端与运营栈。

### 取舍理由

- 推荐方案把风险前置到文档与边界设计里处理，长期成本更低。
- 两种备选方案要么继续积累返工债务，要么过度设计当前 MVP。

## 13. 交接说明（供 flutter-rd-module-splitter 继承）

- 模块拆分必须继承以下全局决策：
  - 本地优先，任务主链路不依赖远端服务
  - 事项持久状态只允许 `active / completed / deleted`
  - 所有关键状态流转必须经应用层 UseCase 并补事件日志
  - Widget / 通知 / 系统入口只消费稳定快照或受控 DTO
  - 展示层统一采用 Riverpod 体系，后续补齐 `riverpod_annotation + riverpod_generator`
  - 结构化数据走 `drift`，轻量配置走 `shared_preferences`
  - 国际化必须走 `gen-l10n`
  - 隐私默认高于展示便利
- 模块拆分阶段只允许在这些全局约束内细化，不得推翻包栈、状态模型或数据真相源。
- 设计相关模块文档必须引用 `D:\Projects\Flutter\screen_note\designs\app.pen`，并在冻结后再进入实现细化。

## 14. 最终结论

- 推荐主方案：以“本地优先 + 分层 feature 架构 + 稳定快照系统桥接 + 未来远端可插拔”的 Flutter 技术基线推进屏记。
- 这是当前最合适路径，因为它直接服务 PRD 的 P0 承诺：可靠、不丢、持续可见、快速录入，同时不会过早把项目拖入同步、鉴权和运营重基础设施。
- 当前不推荐的路径：
  - 继续无基线地追加代码
  - 现在就引入完整云同步/网络主栈
  - 在页面层分散处理状态流转、日志和系统桥接
- 下一步建议：对本技术基线做一次确认；确认后进入设计冻结检查，判断 `app.pen` 是否足以作为后续模块拆分与实现来源。
