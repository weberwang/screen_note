# Mobbin 方向证据补充

## 目的

本文件用于补齐共享设计方向阶段的可追溯灵感证据，避免“只有生成结果，没有参考来源”的问题。这里记录的是方向证据，不是最终设计源替代物；最终共享视觉基线仍以已选定代表图与 `DESIGN.md` 为准。

## 选用的 Mobbin 参考

### 1. Tiimo iOS Plan Flow

- 链接：https://mobbin.com/explore/flows/29cf92cd-18f1-4bcc-8cef-28b3e9a18adf
- 选用原因：
  - 强调“当天计划 / 当前事项”的单日聚焦感。
  - 更接近“减少遗忘焦虑、提升执行聚焦”而不是复杂任务管理。
  - 适合为屏记提供“首页先看最重要的一件事”的结构依据。
- 提炼要点：
  - 首页应优先服务当前最重要事项。
  - 列表是补充，不应压过主任务。
  - 任务处理应有温和、低压的视觉节奏。

### 2. Todoist iOS Calendar / Today Task List 相关屏

- 链接一：https://mobbin.com/explore/mobile/screens/calendar
- 链接二：https://mobbin.com/explore/mobile/app-categories/productivity
- 选用原因：
  - 在 Mobbin 分类里明确出现了 `Stacked List`、`Floating Action Button`、`Goal & Task`、`Home` 这组与屏记高度相关的组织方式。
  - 适合作为“次级事项列表如何快速扫读”的结构参考。
- 提炼要点：
  - 次级任务应采用统一列表行，不适合做成多层卡片拼贴。
  - 快速新增动作可以保持稳定、可见、独立，不必挤进列表区。
  - 时间与状态信息要轻量但可快速扫读。

### 3. Reminders iOS Calendar

- 链接：https://mobbin.com/explore/mobile/screens/calendar
- 选用原因：
  - 该分类中包含 Reminders iOS 的 `Date Picker / Calendar / Scheduling` 相关屏，说明系统原生提醒类产品的日期与提醒表达方式可作为可信参考。
  - 适合为屏记补足“提醒感、日期感、系统原生可信度”的表达依据。
- 提炼要点：
  - 日期/提醒信息应作为任务可信度的一部分，而非装饰文本。
  - 轻量提醒产品更适合明确的日期标签，而不是复杂日历编排。

### 4. Evernote / Note Detail 分类

- 链接一：https://mobbin.com/explore/flows/02d5b9c1-dcca-44a8-97f0-1b3776714c1e
- 链接二：https://mobbin.com/explore/mobile/screens/note-detail
- 选用原因：
  - 虽然屏记不是传统笔记产品，但“把一句事项写下来并快速回看”的心智，与 note-detail 类屏幕有部分重叠。
  - 这组参考主要用于校正“输入/记录感”与“轻量正文承载感”，不用于引入复杂编辑器结构。
- 提炼要点：
  - 保留一句话事项的轻记录感是合理的。
  - 但不能滑向重编辑器或文档工具界面。

## 对当前 3 个方向图的回看

### 方向 A

- 对应参考：
  - Tiimo 的单任务聚焦感
  - Todoist 的任务列表与稳定新增动作
  - Reminders 的时间/提醒可信表达
- 结论：
  - 与 Mobbin 证据最一致。
  - 最符合“锁屏优先、防忘更轻松”的首页心智。
  - 保留为当前设计周期的共享代表性基线。

### 方向 B

- 对应参考：
  - Note Detail 类的“记录感”
  - Evernote / Note UI 的轻文档感
- 结论：
  - 有一定合理性，但便签纸语义偏强，容易把产品理解带向“记事便签”而不是“可靠提醒”。
  - 可作为局部材质灵感，不适合做共享主方向。

### 方向 C

- 对应参考：
  - Todoist / Reminders 的任务效率与日期表达
  - 一部分系统级任务工具的高可读性结构
- 结论：
  - 信息清晰，但整体更像“任务处理器”而不是“锁屏防忘助手”。
  - 适合吸收其状态标签与列表秩序，不适合作为主视觉方向。

## 最终判断

- 当前共享视觉基线使用 `AB` 融合方向：保留 A 的单主任务层级和轻列表结构，吸收 B 的轻微卡片温度，但不继续推进到 `AB2` 精修稿。
- 需要吸收的 Mobbin 结论是：
  - 保留 A 的“单主任务 + 轻列表 + 稳定快速添加”
  - 从 Todoist / Reminders 继续强化时间与状态的扫读效率
  - 仅保留 B 的温和触感，不引入其强便签隐喻
  - 避免 C 的过度工具化倾向

## 对下游产物的影响

- `DESIGN.md` 继续沿用当前方向，但主卡片材质描述更新为“轻微纸感温度”。
- 后续共享 HTML 原型与共享冻结，应明确继承：
  - 单主任务优先
  - 列表行而非卡片堆叠
  - 时间/提醒状态清晰可扫读
  - 全局快速添加稳定存在
  - 主卡片可带轻微温和纸感，但不演变为便签工具
