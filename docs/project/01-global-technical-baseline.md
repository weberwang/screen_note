## 1. PRD 需求理解

### 1.1 产品目标

屏记的核心目标不是替代完整待办系统，而是把“最怕忘的小事”稳定地固定在用户高频可见的位置，并用低输入成本、低打扰和高可靠的方式降低遗忘率。

### 1.2 目标用户

- 主要用户：高频看手机、容易被打断、需要持续视觉提醒的 iPhone 用户。
- 次级用户：已经在用备忘录、聊天助手或系统提醒自我提醒，但对“记了还是会忘”不满的轻量待办用户。
- 非核心用户：需要复杂项目管理、多人协作、审批、甘特图或知识整理的人群。

### 1.3 核心场景

- 3 秒内创建一条重要事项。
- 事项在锁屏、小组件、提醒或系统快捷入口中持续可见。
- 事项过期后仍保留为 `active`，只改变展示标签而不自动终结。
- 删除与恢复链路可追溯，避免误删与误消失。

### 1.4 范围内功能

- 事项创建、编辑、完成、删除、恢复
- 小组件/锁屏快照展示
- 最近完成、最近删除、操作日志
- 本地提醒、基础隐私模式、系统快捷入口

### 1.5 明确不做

- 团队协作与复杂任务管理
- 首发阶段完整 Android 产品
- 自动 AI 建任务
- 大而全跨平台同步体系

## 2. 假设前提与待确认项

### 2.1 假设前提

- 首发 MVP 以单用户、本地优先为主，不依赖登录态即可完成核心事项链路。
- 事项、日志、快照和设置以本地存储为默认真源；iCloud 同步不进入 MVP 核心依赖。
- 通知、小组件、快捷入口都属于能力增强层，任何单项能力失败都不能阻塞事项主链路。
- Android 近期只承接等待名单、调研入口或后续验证，不与 iOS 同步建设完整实现。

### 2.2 待确认项

- 首发是否同时上线 Pro 与试用机制；这会影响付费墙埋点、设置页结构和版本策略。
- iCloud 同步进入哪个阶段；这会影响仓储抽象与冲突处理范围。
- Apple Reminders 协同是导入、单向同步还是双向同步；这会直接改变数据边界与回写策略。
- 锁屏默认布局是单条优先还是允许首次选择 1 条 / 3 条；这会影响共享壳层与展示组件契约。
- Android 等待名单的入口来自 App 内还是外部落地页；这会影响埋点与运营配置边界。

## 3. 业务流程与页面范围

### 3.1 主业务流程

1. 用户进入首页或快速添加入口。
2. 输入一句话事项并保存。
3. 应用层写入任务、日志与稳定快照。
4. Widget/锁屏桥接层读取稳定快照并尝试刷新展示。
5. 用户在 App、系统入口或后续提醒中继续处理事项。
6. 事项进入完成或删除状态，最近记录中心可追溯与恢复。

### 3.2 关键异常流程

- 通知权限拒绝：仅降级提醒，不阻塞事项创建与展示。
- Widget 刷新失败：继续展示最后一次有效快照。
- 事项过期：展示态变为“已过期”，但存储态仍是 `active`。
- 误删：通过最近删除恢复，并补齐恢复日志。

### 3.3 页面与能力范围

推荐以“共享壳层 + 能力域”组织，而不是先按视觉页面拆散：

- 首页 / 任务流：当前待处理事项与快速添加
- 事项编辑能力：标题、备注、时间、隐私、提醒、置顶
- 历史中心：最近完成、最近删除、恢复
- Widget 桥接：快照生成、共享数据、系统回流
- 设置中心：通知、隐私、展示样式、同步、会员入口

## 4. 总体技术架构

### 4.1 推荐方案

采用“本地优先、DDD 按功能分层、系统能力桥接内聚”的 Flutter 架构：

- 目录按 `app/`、`shared/`、`features/<feature>/` 组织。
- 每个 feature 内部按 `presentation / application / domain / data` 分层。
- 事项真源以本地数据库为中心，Widget、通知、快捷入口只消费应用层编排后的稳定结果。
- 所有会改变事项状态或关键字段的动作，只能经由应用层用例完成。

### 4.2 备选方案

- 方案 A：全局平铺目录 + Provider 直接调仓储
- 方案 B：同步优先架构，首期就预留完整云端模型和账号体系

### 4.3 取舍理由

- 方案 A 上手快，但会把状态流转、日志写入、快照刷新等规则散落在页面与 Provider 中，后续极易破坏可靠性。
- 方案 B 对未来扩展友好，但会把 MVP 从“验证锁屏防忘”拉向“搭完整待办底座”，显著增加复杂度和不确定性。
- 本项目最重要的不是横向扩展，而是纵向保证“事项不乱、状态可追、快照稳定”，因此优先选择本地优先与应用层统一编排。

## 5. Flutter 客户端详细技术方案

### 5.1 项目结构

#### 推荐方案

- `app/`：应用入口、路由、启动与全局装配
- `shared/`：主题、通用组件、共享内核
- `features/task_flow/`：事项主链路
- `features/history_center/`：最近完成与最近删除
- `features/widget_bridge/`：快照桥接与系统共享
- `features/settings_center/`：偏好、权限、展示设置
- `features/app_shell/`：共享壳层与系统回流宿主

#### 备选方案

- 单一 `lib/screens + services + models`

#### 取舍理由

- 当前 PRD 已经明确状态、日志、快照、提醒、隐私都属于不同职责域，继续用平铺目录会让边界越来越模糊。

### 5.2 状态管理与依赖注入

#### 推荐方案

基于 `hooks_riverpod + riverpod_annotation` 组织展示层状态与依赖注入：

- 页面与弹层默认使用 `HookConsumerWidget`
- Provider、Notifier、异步装配优先用注解生成
- 页面层只负责输入、展示、导航与副作用触发

#### 备选方案

- `Bloc/Cubit`
- 手写 `ChangeNotifier` / 全局单例

#### 取舍理由

- 现有 `pubspec.yaml` 已明确对齐 Riverpod 3.x 生态，继续沿用能减少额外技术噪音。
- 相比 Bloc，本项目状态更偏本地交互与单机业务流，Riverpod 注解链路更轻。
- 手写单例会破坏测试替换、生命周期管理和 feature 隔离。

### 5.3 路由与共享壳层

#### 推荐方案

基于 `go_router` 维护统一路由，壳层负责顶级导航与系统入口回流，不承载业务判断。

建议的共享壳层方向：

- 首页：主任务流
- 历史：最近完成 / 最近删除
- 设置：偏好、权限、会员、同步
- 快速添加优先作为浮层或上下文入口，而不是单独常驻底栏导航

#### 备选方案

- 原生 `Navigator` 手工管理
- 将每个入口直接做成独立页面树

#### 取舍理由

- 本产品入口多、状态短、回流场景多，用 `go_router` 更适合统一系统入口跳转与深链回流。
- 快速添加更像高频动作而非稳定信息目的地，放入壳层的全局 CTA 更贴合任务导向。

### 5.4 本地数据与仓储

#### 推荐方案

使用 `drift + sqlite3_flutter_libs + path_provider` 作为结构化本地真源：

- `tasks`：事项主表
- `task_events`：状态与关键字段操作日志
- `widget_snapshot_meta`：快照版本、更新时间等桥接元数据
- `settings_preferences`：用户偏好与轻量配置

#### 备选方案

- 仅用 `shared_preferences`
- 直接写 JSON 文件

#### 取舍理由

- `shared_preferences` 不适合可靠维护任务列表、日志链路与筛选恢复。
- JSON 文件对迁移、查询、排序和并发更新不友好。
- `drift` 更适合承接“状态只有三种、过期是派生态、删除可恢复、日志必须完整”的约束。

### 5.5 提醒、小组件与系统能力桥接

#### 推荐方案

对三类系统能力分别封装边界层：

- `notifications/`：提醒调度、权限查询、取消与降级
- `widget_bridge/`：快照投影、共享写入、系统点击回流
- `app/startup` / `app_shell`：深链、Action Button、快捷入口回流

#### 备选方案

- 页面直接操作 `home_widget`、`flutter_local_notifications`

#### 取舍理由

- 这些能力都高度依赖平台条件与失败降级，页面直接接插件会把权限、异常、重试和隐私规则散落到多个位置。
- 边界层统一封装后，更容易补齐 Fake/Mock 与降级测试。

### 5.6 国际化与文案策略

#### 推荐方案

- 全量走 Flutter `gen-l10n`
- 文案资源统一维护在 `lib/l10n`
- 页面、通知、空态、错误提示都通过 `AppLocalizations` 获取

#### 备选方案

- 中文文案先写死，后续再补国际化

#### 取舍理由

- 项目约束已经明确禁止长期硬编码用户可见文案，直接按官方链路落地最省返工。

### 5.7 错误处理与降级

#### 推荐方案

- 领域层输出业务失败类型
- 应用层负责将失败映射为可记录、可观测、可降级的结果
- 展示层只消费“可呈现状态”，不直接处理插件错误码

#### 备选方案

- 页面 `try/catch` 后直接弹 Toast

#### 取舍理由

- 本项目大量失败都不是“不能继续”，而是“能力降级但主链路仍可用”，必须由应用层统一定义行为。

### 5.8 同步与远程扩展预留

#### 推荐方案

- MVP 先保持纯本地数据主链路
- 同步仓储接口提前抽象为可替换端口，但不同步实现不进入首发关键路径
- 若后续进入 iCloud 或 Reminders 协同，再增量引入同步映射、冲突解决和幂等策略

#### 备选方案

- 首发直接接入双向云同步

#### 取舍理由

- 当前 PRD 的核心验证点是“少忘事”和“敢相信”，不是多端一致性。
- 过早同步会把冲突、时序、回写策略、删除语义全部拉进 MVP 风险面。

## 6. 第三方包选型与最佳搭档

### 6.1 状态管理与注解生成

能力域：
推荐主方案：`hooks_riverpod + flutter_riverpod + riverpod_annotation + riverpod_generator`
推荐理由：与现有依赖一致，适合按 feature 拆分 Provider 与副作用装配。
最佳搭档：`flutter_hooks`
备选方案：`Bloc/Cubit`
避免混搭：不要并行引入另一套全局状态框架。
接入注意事项：Provider、Notifier 与异步状态优先走注解生成。
升级/维护风险：Riverpod 大版本升级时需关注生成器与 analyzer 兼容窗口。

### 6.2 本地结构化存储

能力域：
推荐主方案：`drift + sqlite3_flutter_libs + path_provider`
推荐理由：更适合任务、日志、快照元数据与恢复链路。
最佳搭档：`path`
备选方案：`isar` 或 JSON 文件
避免混搭：不要同时维护两套结构化任务真源。
接入注意事项：数据库表结构需直接映射三态规则与软删除策略。
升级/维护风险：schema 变更要配套迁移与回归测试。

### 6.3 提醒与时区

能力域：
推荐主方案：`flutter_local_notifications + timezone + flutter_timezone`
推荐理由：覆盖本地提醒、重复调度与时区感知。
最佳搭档：应用层提醒调度服务
备选方案：只做系统通知占位
避免混搭：不要把时间计算分散到页面与插件调用处。
接入注意事项：通知失败只能视为能力降级。
升级/维护风险：iOS 权限策略与系统限制需要持续验证。

### 6.4 Widget 与锁屏桥接

能力域：
推荐主方案：`home_widget`
推荐理由：对桌面/锁屏 Widget 桥接较直接，适合稳定快照写入。
最佳搭档：`widget_bridge` 封装层
备选方案：直接在平台侧手写共享逻辑
避免混搭：不要让多个页面各自写 Widget 共享数据。
接入注意事项：始终以稳定快照而非临时查询结果供给 Widget。
升级/维护风险：系统刷新时机不可控，必须持续保留最后有效快照。

### 6.5 模型与序列化

能力域：
推荐主方案：`freezed + freezed_annotation + json_serializable + json_annotation`
推荐理由：统一不可变模型、值相等与 JSON 映射。
最佳搭档：`build_runner`
备选方案：手写实体与 DTO
避免混搭：不要长期保留手写样板和注解生成并存。
接入注意事项：`domain/` 与 `data/` 下实体优先使用注解定义。
升级/维护风险：生成链升级要保持同一兼容窗口。

### 6.6 网络与安全存储

能力域：
推荐主方案：`dio + retrofit + flutter_secure_storage`
推荐理由：虽然 MVP 可不启用远程依赖，但为未来 iCloud 之外的服务协同保留统一接入方式。
最佳搭档：统一错误映射与拦截器层
备选方案：原生 `http`
避免混搭：不要未来再补一套平行网络层。
接入注意事项：只有真正进入远程能力时再打开接口实现。
升级/维护风险：若长期不用，需避免“预埋但无人维护”的空壳接口扩散。

## 7. 后端协作与接口契约要求

### 7.1 推荐方案

MVP 默认不依赖自建后端；但为未来同步、付费或跨设备增强保留如下契约基线：

- 时间统一使用 ISO 8601 与明确时区
- 删除语义保持软删除，不引入“远端硬删、本地软删”双标准
- 若未来接入同步，所有状态变更接口必须支持幂等
- 若未来接入分页历史或日志拉取，排序字段应明确以服务端时间还是设备时间为准

### 7.2 备选方案

- 先不上任何远程契约，等要做同步时再补

### 7.3 取舍理由

- 当前不做远程实现是合理的，但完全不定义协作边界会让后续接云能力时返工仓储、日志与状态语义。

## 8. 数据与安全方案

### 8.1 推荐方案

- 事项正文、隐私标记、提醒配置、删除时间等结构化数据落本地数据库
- 敏感令牌或未来凭证进入 `flutter_secure_storage`
- Widget、锁屏、通知仅读取最小必要展示字段
- 隐私事项默认不在锁屏、通知文本与系统快捷入口中泄露正文

### 8.2 备选方案

- 直接把所有字段都投影到 Widget 与通知层

### 8.3 取舍理由

- 本产品信任底座就是“可见但不泄密”，因此边界必须先严后松。

## 9. 埋点、监控与运营支持

### 9.1 推荐方案

围绕“激活、可靠性、留存、商业化”四条主线设计埋点：

- 激活：`task_created`、`widget_config_started`、`widget_config_completed`
- 可靠性：`task_restored`、`widget_snapshot_refresh_failed`、`notification_schedule_failed`
- 留存：`task_completed`、`quick_add_used`
- 商业化：`pro_paywall_viewed`、`pro_trial_started`、`pro_subscribed`

监控上优先覆盖：

- 崩溃与严重异常
- 快照刷新失败
- 通知调度失败
- 软删除恢复失败

### 9.2 备选方案

- 只接基础崩溃监控，不做业务事件分类

### 9.3 取舍理由

- 本产品的成败高度依赖“用户是否真的看见、是否真的信任”，仅看 crash 不足以发现核心问题。

## 10. 测试与质量保障

### 10.1 推荐方案

- 单元测试：状态流转、过期派生、排序、恢复与日志写入
- 数据层测试：仓储与 Drift 映射、软删除、快照投影
- Widget/页面测试：首页、历史中心、设置中心、隐私态与空状态
- 集成测试：通知拒绝、Widget 刷新失败、系统入口回流降级

### 10.2 备选方案

- 只做页面冒烟测试

### 10.3 取舍理由

- 对“事项不消失、恢复可用、快照稳定”这类约束，仅靠页面冒烟很难兜住。

## 11. 实施计划与里程碑

### 11.1 推荐主线

1. 全局技术基线确认
2. 共享设计方向、公共壳层与 DESIGN.md 确认
3. 共享设计冻结
4. 项目初始化与全局 bootstrap 基线
5. 模块拆分与 `impl.md`
6. 模块设计冻结
7. 串行实现：任务流 -> Widget 桥接 -> 历史中心 -> 设置中心 / 壳层完善

### 11.2 备选方案

- 先写功能代码，后补设计冻结与模块文档

### 11.3 取舍理由

- 当前项目对设计冻结、模块文档与实现顺序都有明确流程约束，逆序推进会导致返工。

## 12. 风险清单与替代方案

### 12.1 技术风险

- Widget 与锁屏刷新不稳定，可能导致“有记录但没展示”
- 过早加入同步会放大冲突与删除语义复杂度
- 多入口回流若未统一编排，容易出现状态不一致

### 12.2 协作风险

- 若共享壳层未提前定下来，后续页面与导航会反复调整
- 若 Product Design brief 不明确，模块拆分会误判信息密度与首屏任务

### 12.3 替代方案

- 若首发不做 Pro，可先把会员入口降为占位信息结构，不接订阅实现
- 若同步暂缓，先把仓储接口抽象为本地实现 + 未来远程实现占位

## 13. 模块拆分交接说明

后续 `flutter-rd-module-splitter` 必须继承以下全局决策：

- 主平台是 iOS，当前设计基线视口为 `390 x 844 px`
- 事项持久状态只允许 `active`、`completed`、`deleted`
- 过期仅是展示派生状态，不允许落库
- 所有关键状态变更必须走应用层用例并写日志
- Widget、通知、系统入口都只能消费稳定结果，不直接承载复杂业务推导
- 快速添加属于高频动作，默认优先级高于二级设置入口
- 共享壳层应服务任务清晰度，不应为“看起来完整”而过度扩导航

## 14. 最终结论

### 推荐方案

先以“本地优先 + 按功能分层 + 系统能力桥接内聚”的 Flutter 基线推进，优先把事项真源、日志、快照、提醒与隐私边界做稳，再在共享设计冻结后进入模块拆分和串行实现。

### 备选方案

- 同步优先的云端待办架构
- 平铺目录的轻脚手架实现

### 取舍理由

- 同步优先会把 MVP 复杂度抬高到当前验证阶段不需要的程度。
- 平铺脚手架虽然启动更快，但很难支撑“锁屏可靠展示 + 状态可追溯 + 降级不阻塞”的长期要求。

### 当前建议

- 这一阶段确认全局技术基线成立。
- 下一阶段应进入 Product Design brief 播放确认，并同步确认共享公共壳层方向。
